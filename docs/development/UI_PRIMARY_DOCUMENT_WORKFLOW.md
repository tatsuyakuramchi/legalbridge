# UI主導の文書作成フロー

## 採用方針

文書作成の主体を `Backlog -> UI / DB` に寄せる。

役割分担は次のとおり。

- `Backlog`
  - 案件の管理台帳
  - Local 起動の基準情報
  - ステータス管理
  - 誰が / 何を / いつまでに / どの相手方で進めるか
- `UI`
  - 文書編集
  - 入力補完
  - 下書き保存
  - 最終確認
  - 文書生成
- `DB`
  - 文書作成データの正本
  - マスタ
  - 生成履歴
  - ワークフロー履歴
- `Slack`
  - 任意の受付チャネル
  - 通知、承認、押印などの対人アクション
  - 未接続でも Backlog / UI / DB の主線は止めない

## 基本フロー

1. Backlog に課題を作成する
2. Local が Webhook または Polling で課題変化を検知する
3. UI で課題キーを指定して編集画面を開く
4. Backlog + DB + マスタから初期値を補完する
5. UI で編集・保存する
6. UI から Drive に文書を出力する
7. Backlog のステータスを更新する
8. 必要に応じて Slack 通知や承認・押印へ進める

補足:

- Cloud Run 受付では `Backlog 起票を最優先` にし、DB 保存や法務チャンネル通知は後段の任意処理として扱う
- そのため `SLACK_BOT_TOKEN` や `SLACK_LEGAL_CHANNEL` が未設定でも、Backlog / UI / DB の主線は継続できる

## Backlog の役割

Backlog は `詳細付きの課題管理` として扱う。

### Backlog に残すもの

- 契約種別
- 相手方名
- 相手方住所
- 相手方代表者
- 契約書番号
- 契約日
- 希望期限
- 契約概要
- 特記事項
- 備考
- 親課題キー
- 紐付けライセンス課題キー

### Backlog に残すときの基準

- 誰が見ても案件を識別できる項目
- ステータス管理と検索に必要な項目
- 後続課題へ渡す参照キー
- Slack 起案の時点で確実に埋められる項目

### Backlog に残さない考え方

- 文書レイアウトのためだけに必要な値
- マスタ参照で補完できる値
- UI で最終確認してから決める値
- 明細、素材、金銭条件のような増減しやすい値

### Backlog に残さないもの

- 口座情報
- 自社情報の詳細
- 文書ごとの細かな表示値
- ライセンス個別条件
- 金銭条件
- 素材情報
- 派生計算値
- 生成済み文書の編集値

これらは DB と UI に寄せる。

## UI の役割

UI は `文書作成の正本編集画面` として扱う。

### UI で行うこと

- Backlog のヘッダ確認
- DB マスタ補完
- 文書ごとの詳細入力
- 文書種別別の表示切替
- 下書き保存
- 生成前確認
- 文書生成

### UI に必要な性質

- 文書種別ごとに項目が変わる
- 下書きを DB に保存できる
- 再編集できる
- Drive 出力後に Backlog ステータス更新できる

## DB の役割

DB は文書作成値の正本とする。

### 主な保持先

- `IssueWorkflow.documentDraft`
  - 文書編集下書き
- `Vendor`
  - 相手方マスタ
  - 電話番号
  - メール
  - 銀行名 / 支店名 / 口座種別 / 口座番号 / 口座名義(カナ)
  - 適格請求書発行事業者フラグ
  - 適格請求書登録番号
  - 基本契約参照
- `Staff`
  - 自社担当者マスタ
- `IssueWorkflow.generatedDocuments`
  - 生成履歴

## 文書種別ごとの考え方

### NDA

- Backlog
  - 相手方
  - 契約概要
  - 契約日
  - NDA 条件
- UI
  - 文書表示の最終調整

### 業務委託基本契約

- Backlog
  - 相手方
  - 契約概要
  - 契約日
- UI
  - 口座情報
  - 自社情報
  - 文書表示値
  - マスター契約参照有無に応じた発注書生成制御

### ライセンス契約

- Backlog
  - 相手方
  - 契約概要
  - 契約日
  - 原著作物など基本情報
- UI
  - 基本契約の最終調整
  - 口座情報
  - インボイス制度関連
  - 自社情報
- 個別条件は別課題

補足:

- ライセンス契約の頭書きは `業務委託基本契約` と同じ粒度にそろえる
- 相手方は `住所 / 商号 / 代表者 / 電話番号 / メール`
- 自社は `住所 / 商号 / 代表者`
- 特約・特記事項は固定文を持たず、編集値をそのまま表示する

### 個別利用許諾条件

- Backlog
  - 親ライセンス課題キー
  - 契約概要
  - 許諾開始日
- UI
  - 素材情報
  - 金銭条件
  - 許諾範囲
  - 文書生成値の正本

### 売買契約

- Backlog
  - 相手方
  - 契約概要
  - 契約日
- UI
  - 売買条件の詳細

## 補完ルールの補足

- 発注書系とライセンス契約系は、Backlog 未入力項目を `Vendor` マスタから補完する
- 補完対象には、電話番号、メール、口座情報、インボイス情報、基本契約参照を含める
- `発注書のみ` で `MASTER_CONTRACT_REF` が空の場合は、発注書テンプレートに `terms_spot_2026.html` を連結して 1 つの PDF として生成する
- `/法務検索` は Backlog の案件情報を正本として検索し、Cloud Run 側からでも後続申請へ進める
- 納品リクエストは発注明細候補を DB から取れない場合でも、親課題キーを保持したまま `item_no` 手入力で起票できる
- 利用許諾料計算と個別利用許諾条件は、親案件の検索と起票を Backlog ベースで継続できる

## 今後の実装方針

### 第1段

- Backlog に送る項目を最小限に絞る
- UI で詳細編集する前提にする
- Slack 起票ではヘッダだけを Backlog に送る
- UI の初回読込で Backlog + DB + 既存下書きを統合する

### 第2段

- 文書種別ごとに UI を強化する
- ライセンス系は `基本契約` と `個別利用許諾条件` を分離する

### 第3段

- Backlog ステータス更新を UI 生成処理に組み込む
- 承認・押印・後続申請と接続する

## まとめ

今後の主線は次のとおり。

`Backlog -> UI / DB -> Drive -> Backlog状態更新`

この方針により、

- Backlog の属性数を抑えられる
- 文書種別ごとの UI 最適化ができる
- DB を正本にできる
- 実務変更に強くなる
