# Data Allocation Policy

## 目的

Slack をハブにせず、Backlog / Local / DB の責務を分離したうえで、Backlog 情報から Local が動ける構成にする。

## 基本方針

- `Backlog` を案件管理の正面入口にする
- `Local` は Backlog の課題情報を読んで UI 表示、文書生成、状態反映を行う
- `DB` はマスタ、履歴、明細、計算結果、下書き、ファイル参照の正本を保持する
- `Slack` は受付や通知などの外部チャネルとして扱い、Backlog / Local / DB のハブにしない
- `アプリ` は採番、整形、計算、固定値補完、テンプレート変換を担当する

## Backlog 主体で扱う文書

以下は単票に近く、主要入力を Backlog カスタム属性で持つ。

- NDA
- 業務委託基本契約
- ライセンス契約
- 個別利用許諾条件
- 売買契約（当社買手）
- 売買契約（当社売手・標準）
- 売買契約（当社売手・保証金掛け売り）

### Backlog に置くもの

- 契約日
- 契約期間
- 相手方名
- 相手方住所
- 相手方代表者
- 特約
- 備考
- 文書固有の主要条件

### DB に置くもの

- 相手方マスタ
- 担当者マスタ
- 既存契約参照
- 承認履歴
- 押印履歴
- 生成文書 URL

## 例外的に DB 主体で扱う文書

以下は明細の繰り返しが強く、Backlog カスタム属性だけでは維持しにくい。

- 発注書
- 企画発注書
- 納品リクエスト
- 検収書
- 支払通知書
- ロイヤリティ報告書
- レベニューシェア報酬計算書

### Backlog に置くもの

- 案件ヘッダ
- 相手方
- ステータス
- 支払条件の概要
- 納期や基準日などの主要日付

### DB に置くもの

- 明細行
- 金額内訳
- 差戻し・変更履歴
- 集計値
- 文書 URL
- Slack スレッド情報

## 発注書系の扱い

### 発注書

- Backlog にはヘッダ中心を置く
- 明細が 1 行で済む場合は Backlog 単独でも許容
- 複数明細は DB を正本とする

### 企画発注書

- Excel / CSV を明細ソースとして許容する
- Backlog には案件ヘッダと進行状態のみを保持する
- detailText 組み立てはアプリ側で行う

## 個別利用許諾条件の扱い

- Backlog 主体で扱う
- ただし金銭条件の繰り返し構造が増える場合は DB テーブルへ分離する
- 現時点では主要条件を Backlog に置き、複雑な条件行のみ将来 DB 化する

## 採用ルール

- 単票文書は `Backlog 主体`
- 繰り返し明細文書は `DB 主体`
- Backlog カスタム属性は人間が案件管理で読む値を優先する
- DB は文書再生成に必要な完全情報を保持する
- アプリは Backlog の不足分を DB から補完する

## 実装上の前提

- Backlog 課題が存在すれば、Slack 未接続でも Local は処理を継続できる
- 文書種別ごとに Backlog 課題タイプを分ける
- 自動生成対象は単票文書を優先する
- 発注書系は起票後に詳細取込を行う
- Slack 通知は任意であり、未設定時はスキップしてよい
