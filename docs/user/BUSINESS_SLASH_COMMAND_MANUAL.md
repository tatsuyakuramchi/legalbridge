# 事業部向け Slack スラッシュコマンド / 申請マニュアル

## 目的

このマニュアルは、事業部メンバーが Slack から法務依頼を送るときに、

- どのスラッシュコマンドを使うか
- どの申請方法を選ぶか
- 何を入力すればよいか
- 送信後に何を確認すればよいか

を迷わず判断できるようにするための利用者向け手順書です。

対象は、現在実装されている Slack コマンドと申請フォームです。

## まず覚えること

- 新しい法務依頼を出すときは `/法務依頼`
- 過去案件や親課題を探すときは `/法務検索`
- 進捗確認は `/法務ステータス LEGAL-123`
- 直近案件の一覧確認は `/法務一覧`

基本の流れは次のとおりです。

1. Slack で依頼を送る
2. Backlog に課題が作成される
3. 必要に応じて法務チーム通知や内部記録が行われる
4. 法務担当が UI で内容を確認、補完、文書生成する
5. 必要に応じて承認、押印、納品、利用許諾料計算などの後続申請へ進む

補足:

- Slack 受付は Cloud Run 側でも動かせるため、ローカルアプリ停止中でも `/法務依頼` と `/法務検索` の受付を継続できる構成です
- Cloud Run 側ではまず Backlog 起票を優先し、ローカル DB 保存や法務チャンネル通知は設定がある場合のみ実施します

## スラッシュコマンド一覧

### `/法務依頼`

新しい法務依頼を起票するコマンドです。  
最初に使うメインの入口です。

### `/法務検索`

親案件や過去案件を探すコマンドです。  
次のような場面で使います。

- 納品リクエストの元になる発注書を探したい
- 利用許諾料計算の対象になるライセンス案件を探したい
- 課題キーを忘れたので案件名や相手方名で探したい

検索語には次のようなものが使えます。

- 課題キー
- 相手方名
- 案件名
- 発注先名
- 原著作物名

現在の `/法務検索` はローカルDB検索ではなく、Backlog の案件情報を検索します。

- 発注書
- 企画発注書
- ライセンス契約
- 個別利用許諾条件

検索結果から、そのまま次の後続申請へ進めます。

- 納品リクエスト
- 個別利用許諾条件
- 利用許諾料計算

### `/法務ステータス LEGAL-123`

課題キーを指定して、その案件の現在ステータスを確認するコマンドです。

例:

```text
/法務ステータス LEGAL-123
```

### `/法務一覧`

直近 8 件の法務案件を一覧で確認するコマンドです。  
自分が今追っている案件のざっくり確認に向いています。

## 共通の入力ルール

どの申請でも、まず次の共通項目を入れます。

| 項目 | 使い方 |
|---|---|
| 保存先Drive | 生成文書や関連資料の保存先フォルダを選びます |
| 登録番号 | 個人は執筆者登録番号、法人は国税の登録番号を入力します。法務相談では未確定なら空欄でも受け付けます |
| 相手方名 | 正式名称を入力します。法務相談では相談先やレビュー対象があれば入力します |
| 相手方所在地（住所） | 契約相手の住所を入力します。法務相談では任意です |
| 相手方代表者 | 法人相手なら代表者名を入力します。個人は通常不要です |
| 概要（件名） | 何の依頼か一目で分かる件名を書きます |
| 文書作成希望完了日 | 希望納期がある場合に入力します |
| 備考 | 運用メモ、口頭合意、社内事情などを補足します |
| 補足・参考資料 | 契約背景、既存契約との関係、前提条件などを書きます |

入力のコツ:

- 件名は短く具体的に書く
- 金額条件や例外条件がある場合は備考か補足に必ず残す
- 迷う項目は空欄のままにせず、分かる範囲を補足に書く
- 課題キーが返ってきたら必ず控える
- 他社文書レビュー依頼や法務相談は、レビューしてほしい論点を補足欄へまとめる

## 申請方法の選び方

### 1. 法務相談

他社文書のレビュー依頼や、契約化前の法務相談を送りたいときに使います。

主な入力項目:

- 相手方・相談先
- 相談概要
- 相談背景・補足
- 相談内容・レビュー観点

こんなときに使う:

- 他社から受領した契約書やNDAのレビューを依頼したい
- 文書化が必要かどうかを先に相談したい
- 契約前提、進め方、リスクの見立てだけ先に確認したい

ポイント:

- 登録番号、相手方、住所が未確定でも起票できます
- どこを見てほしいか、何が心配かを補足欄に書くと法務側で着手しやすくなります

### 2. NDA

秘密保持契約を作成したいときに使います。

主な入力項目:

- 契約日
- 秘密保持の目的
- 契約期間
- 秘密保持期間
- 管轄裁判所

こんなときに使う:

- 商談前に相手先と情報開示する
- 商品化検討、共同検討、資料開示の前提を整えたい

### 3. 業務委託基本契約

委託先との基本契約を作りたいときに使います。

この申請では作成方式を選びます。

- `基本契約 + 発注書`
- `発注書のみ`

使い分け:

- 継続的な委託関係を作るなら `基本契約 + 発注書`
- 既に基本契約があり、個別発注だけ進めたいなら `発注書のみ`

主な入力項目:

- 契約日
- 契約期間
- 管轄裁判所
- 業務概要・前提情報
- 発注案件名
- 発注概要

注意:

- `発注書のみ` を選ぶと、実際の起票種別は発注書として扱われます
- 発注書の明細は Slack ではなく、後で UI 側で補完します
- マスター契約参照が空のまま発注書を生成する場合は、スポット取引条件書が同じ PDF の後ろに連結されます

### 4. ライセンス契約

ライセンスの基本契約を作りたいときに使います。

この申請では作成方式を選びます。

- `基本契約 + 個別利用許諾条件`
- `個別利用許諾条件のみ`

使い分け:

- 新規案件で基本契約から作るなら `基本契約 + 個別利用許諾条件`
- 既存の基本契約があり、個別条件だけ必要なら `個別利用許諾条件のみ`

主な入力項目:

- 契約日
- 原著作物
- 原著作者
- クレジット表記
- 承継覚書日付
- 契約期間
- 管轄裁判所

注意:

- `個別利用許諾条件のみ` を選ぶと、実際の起票種別は個別利用許諾条件として扱われます
- ライセンス基本契約の頭書きは、業務委託基本契約に近い当事者情報と支払先情報で表示されます
- 特約・特記事項は固定文ではなく、案件ごとの入力内容がそのまま反映されます

### 5. 個別利用許諾条件

ライセンス案件ごとの個別条件を申請するときに使います。  
次の 2 通りの入り方があります。

- ライセンス契約の `個別利用許諾条件のみ` を選ぶ
- ライセンス契約の受付完了後に表示される `個別利用許諾条件を作成` ボタンを押す

主な入力項目:

- 親ライセンス課題キー
- ライセンス種別名
- 原著作物
- 許諾開始日
- 許諾地域・言語
- 計算方式表示
- 料率表示
- 支払条件表示
- MG/AG表示

必要に応じて追加する項目:

- 素材番号
- 素材名
- 素材権利者
- 監修者
- 金銭条件1
- 金銭条件2
- 金銭条件3

使い方のポイント:

- 親ライセンス課題キーは必ず正しいものを使う
- 金銭条件が複数ある場合は条件ごとに分けて入力する
- 詳細が固まっていない項目は、補足に考え方だけでも残す

### 6. 売買契約（当社買手）

当社が買手側になる売買契約で使います。

主な入力項目:

- 契約日
- 商品範囲
- 納入場所
- 検収期間
- 支払条件概要
- 保証期間
- 管轄裁判所

### 7. 売買契約（当社売手・標準）

当社が売手側になる標準的な売買契約で使います。

主な入力項目:

- 契約日
- 商品範囲
- 支払条件概要
- 月末締め日
- 支払期日
- 支払方法
- 保証期間
- 管轄裁判所

### 8. 売買契約（当社売手・保証金掛け売り）

保証金条件がある売買契約で使います。

主な入力項目:

- 契約日
- 商品範囲
- 支払条件概要
- 月末締め日
- 支払期日
- 支払方法
- 保証金額
- 保証金補充期限
- 保証期間
- 管轄裁判所

### 9. 発注書

通常の発注書を作るときに使います。  
次の 2 通りの入り方があります。

- `/法務依頼` で `業務委託基本契約` を選び、作成方式で `発注書のみ` を選ぶ
- 既存フローに沿って法務担当から案内された発注書導線を使う

主な入力項目:

- 案件名
- 発注概要

注意:

- 明細は Slack では入れません
- 明細は管理 UI または CSV で補完します
- マスター契約参照が未設定のときは、スポット取引条件書を後ろに連結した 1 ファイル PDF で発行されます
- 発注書は依頼承諾から 3 営業日が最短目安です

### 10. 企画発注書

大量明細や企画明細を前提とする発注書で使います。

主な入力項目:

- 発注日
- 案件名
- マスター契約参照

注意:

- 明細は XLSX / CSV 取込を前提にしています
- Slack ではヘッダだけを入力します

### 11. 納品リクエスト

発注書または企画発注書の後続申請です。  
単独で課題キーを手打ちするのではなく、次の方法を使う前提です。

1. 発注書や企画発注書の受付完了メッセージを開く
2. `納品リクエストを作成` ボタンを押す

または、

1. `/法務検索` で親案件を探す
2. 対象案件から導線を開く

主な入力項目:

- 親課題キー
- 対象明細
- 今回納品金額
- 納品備考

使い方のポイント:

- 対象明細が出る場合は該当明細を選ぶ
- 金額が確定していない場合も、備考で状況を残す

通知:

- 発注明細の納期を基準に、1週間前、3日前、当日、期限超過後は毎日 Slack で通知されます
- 通知は案件スレッドに集約され、申請者と上長IDがメンションされます

### 12. 利用許諾料計算（製造ベース）

ライセンス案件に紐づく製造実績の登録に使います。  
個別利用許諾条件の受付完了後に表示される `利用許諾料計算（製造）` ボタンから進むのが基本です。

主な入力項目:

- 紐付けライセンス課題キー
- 製品名
- 版
- 製造完了日
- 製造数量
- 基準価格（税抜）
- サンプル数

向いているケース:

- 製造完了日を起点に支払期限や報告期限を管理する案件

### 13. 利用許諾料計算（売上報告ベース）

ライセンス案件に紐づく売上報告型の計算に使います。  
個別利用許諾条件の受付完了後に表示される `利用許諾料計算（売上報告）` ボタンから進むのが基本です。

主な入力項目:

- 紐付けライセンス課題キー
- 対象商品・報告単位名
- 報告対象期間開始
- 報告対象期間終了
- 売上高・正味売上高
- 受領額
- 販売数量

向いているケース:

- 売上報告期間の終了を起点に精算する案件
- 再許諾料や受領額ベースで計算する案件

## 送信後に見るポイント

送信後は、Slack の指定チャンネルに受付メッセージが返ります。必ず次を確認してください。

- Backlog の課題キー
- Backlog を開くボタン
- 後続申請ボタンの有無
- 自分へのメンション
- 必要に応じて上長へのメンション

主な後続導線:

- 発注書 / 企画発注書:
  `納品リクエストを作成`
- ライセンス契約:
  `個別利用許諾条件を作成`
- 個別利用許諾条件:
  `利用許諾料計算（製造）`
  `利用許諾料計算（売上報告）`

## ステータス確認の見方

案件の進捗は `/法務ステータス` で確認します。  
Backlog 側では、主に次のステータスで進みます。

- 文書生成依頼
- 草案
- 草案
- 承認待ち
- 相手方確認待ち
- クラウドサイン送信準備
- 押印依頼中
- 締結済
- 完了
- 破棄

## 迷ったときの判断基準

- 秘密保持なら `NDA`
- まず相談したい、他社文書レビューを依頼したいなら `法務相談`
- 委託先との基本関係を作るなら `業務委託基本契約`
- 既存委託契約の個別発注だけなら `発注書のみ`
- 著作物利用の基本契約なら `ライセンス契約`
- ライセンス案件の個別条件だけなら `個別利用許諾条件`
- 発注後の納品受付なら `納品リクエスト`
- ライセンス精算の起点登録なら `利用許諾料計算`

## よくある注意点

- 課題キーは指定チャンネルの受付投稿に出るため、必ず控えてください
- 納品リクエストや利用許諾料計算は、親案件を検索または後続ボタンから開く前提です
- 発注書や企画発注書の明細は Slack では完結しません
- 最終的な文書内容は、法務担当が UI で確認・補完した内容が正本になります

## 申請者向けのおすすめ運用

1. まず `/法務依頼` で起票する
2. 返ってきた課題キーを控える
3. 必要なら `/法務ステータス` で進捗を見る
4. 後続申請が必要な案件は、受付メッセージのボタンから続けて申請する
